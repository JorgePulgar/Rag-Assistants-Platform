# CODING CONVENTIONS

Style and structure rules. Apply to all code in the project, whether
written by hand or generated by Claude Code.

## Python

### Formatting
- **Black** as formatter (line length 100).
- **isort** for import ordering.
- **ruff** as linter (minimal configuration, default rules).

### Typing
- **Type hints are mandatory** on every public function (full signature).
- On private functions (`_prefix`), types are recommended but not required.
- Pydantic models for anything crossing the HTTP boundary (request/response).
- SQLAlchemy models stay separate from Pydantic schemas — never mix.

### Naming
- `snake_case` for functions, variables, modules.
- `PascalCase` for classes.
- `UPPER_SNAKE` for module-level constants.
- Descriptive names; no abbreviations except for established conventions
  (`db`, `id`, `ctx`).

### Module structure
- One file = one responsibility.
- FastAPI routers (`api/`) contain no business logic — they delegate to
  `services/`.
- Services know nothing about HTTP — they accept and return pure Python
  types or Pydantic models, never `Request`/`Response`.
- Clients (`clients/`) are thin wrappers over external SDKs; they hide
  the SDK API behind an internal interface.

## TypeScript / React

### Formatting
- **Prettier** with defaults (2 spaces, semicolons, single quotes).
- **ESLint** with `eslint-config-react-app`.

### Typing
- `strict: true` in `tsconfig.json`.
- Explicit types for component props (`interface Props { ... }`).
- `type` vs `interface`: use `interface` for props and public object
  shapes, `type` for unions and utility types.

### Components
- One component per file.
- Function components + hooks. No class components.
- Custom hooks start with `use` (React rule).
- File name = component name (`AssistantList.tsx` exports `AssistantList`).

### State management
- Local React state by default.
- Contexts only for genuinely global state (selected assistant, assistant
  list). **Redux is not used.**

## Language

- All code, comments, identifiers, UI strings, log messages, commit
  messages, and documentation are written in **English**.
- User data (assistant names, instructions, document content, chat
  messages) can be in any language — that's data, not UI.
- The only Spanish content in the repository is the secondary `README.es.md`
  (Spanish courtesy version) and any demo fixtures intentionally written
  in Spanish to showcase multilingual support.

## Error handling

### Backend
- **Fail fast**: early Pydantic validation. If something arrives wrong,
  return 4xx immediately.
- **Specific exceptions**: the backend defines its own exceptions
  (`AssistantNotFoundError`, `IngestionError`, `RetrievalError`) in
  `app/exceptions.py`.
- **Global exception handlers** in `main.py`: map domain exceptions to
  HTTP status codes.
- **Never** `except Exception: pass`. If you do not know what to do with
  an error, let it propagate.
- **Azure errors**: capture, log, and re-raise as 502 Bad Gateway with a
  generic message to the client.

### Frontend
- shadcn `toast` for recoverable errors.
- React error boundaries for catastrophic errors (full-page "something
  went wrong" with a reload button).

## Logging

- Backend: Python standard `logging`, configured in `main.py`.
- Default level: `INFO`. `DEBUG` is enabled via `LOG_LEVEL=DEBUG`.
- Format: `%(asctime)s [%(levelname)s] %(name)s: %(message)s`.
- **We always log**:
  - Document ingestion start and end (with chunk_count and duration).
  - Retrieval query (with actual top_k returned and scores).
  - Azure errors (endpoint, status, message).
- **We never log**: API keys, full document contents, full user messages
  (truncate to the first 80 characters for traceability).

## Tests

- **pytest** in the backend.
- Tests in `backend/tests/` with `test_*.py` naming.
- Fixtures in `backend/tests/fixtures/`.
- One test per case, descriptive naming:
  `test_retrieval_returns_empty_when_no_docs_for_assistant`.
- The critical tests listed in `CONSTITUTION.md` are mandatory.
- We do not mock Azure Search in isolation tests; we use a real instance
  with a `test-` index prefix. If Azure is unavailable, those tests are
  skipped via `pytest.mark.skipif`.

## Commits

Convention `type(scope?): message`:

- `feat(backend): add assistant CRUD endpoints`
- `fix(rag): handle empty retrieval result`
- `refactor(frontend): extract MessageBubble from ChatView`
- `docs: update README with setup instructions`
- `test(isolation): add cross-assistant retrieval test`
- `chore: bump dependencies`

One commit = one coherent change. **Never** group unrelated changes into
the same commit.

## Comments

- Docstrings on every public function (`"""Short description. Args: ...
  Returns: ..."""`).
- Inline comments only to explain **why**, not **what**.
  - Bad: `# increment i by 1`
  - Good: `# add 1 because Azure Search indexes pages starting from 1,
    not 0`
- `# TODO:` only with justification and (when applicable) an issue/ticket
  reference. `# FIXME` is never left without an owner.

## Dependencies

- Backend: `requirements.txt` + `requirements-dev.txt`.
- Frontend: `package.json` with pinned versions (no `^` or `~`).
- We do not add a new dependency without justifying why nothing already
  present does the job. Every dependency is attack surface and maintenance.

## Secrets

- Zero secrets in code. Zero in commits (verify with `git diff --cached`
  before committing).
- `.env` is in `.gitignore`.
- `.env.example` lists every variable name with comments describing what
  goes in each (no real values).
